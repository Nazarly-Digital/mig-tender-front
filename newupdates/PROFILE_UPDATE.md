# Редактирование профиля пользователя

Два новых endpoint'а для изменения профиля:

1. `PATCH /api/v1/auth/me/` — любой авторизованный пользователь правит **свой** профиль.
2. `PATCH /api/v1/admin/users/{id}/` — админ правит **любого** пользователя.

Оба используют одну и ту же серверную модель данных и общий набор полей.

---

## 1. Поля, которые можно изменить

| Поле            | Тип     | Доступно кому                   | Примечание                           |
| --------------- | ------- | ------------------------------- | ------------------------------------ |
| `first_name`    | string  | все роли                        | `blank=true` разрешён                |
| `last_name`     | string  | все роли                        | `blank=true` разрешён                |
| `email`         | string  | все роли                        | уникальный, проверка коллизий         |
| `inn_number`    | string  | broker (для dev/admin — тоже, но поле редко заполнено) | 10 или 12 цифр, валидация чексуммы ИНН, уникальный |
| `phone_number`  | string  | broker (тихо игнорируется, если у пользователя нет `broker`) | хранится в `Broker.phone_number`    |
| `company_name`  | string  | developer (тихо игнорируется, если нет `developer`) | хранится в `Developer.company_name` |
| `is_active`     | boolean | **только** admin через `/admin/users/{id}/` | админ не может деактивировать сам себя |

Если не передать ни одного поля — **400**.

Роль (`role`), пароль и флаги `is_staff`/`is_superuser` через этот endpoint **не меняются**.
Пароль — только через существующий `/auth/change-password/`.

---

## 2. `PATCH /api/v1/auth/me/` (self-update)

**Доступ:** авторизованный пользователь с `is_active == true`. Неактивный → 403.

**Пример запроса (броker правит телефон и имя):**

```http
PATCH /api/v1/auth/me/
Authorization: Bearer <jwt>
Content-Type: application/json

{ "first_name": "Анна", "phone_number": "+79990001122" }
```

**Пример запроса (девелопер меняет название компании):**

```json
{ "company_name": "ООО Ромашка" }
```

**Ответ 200:** полный объект `me` (тот же формат, что и `GET /auth/me/`):

```jsonc
{
  "id": 7,
  "email": "broker@example.com",
  "first_name": "Анна",
  "last_name": "Иванова",
  "is_active": true,
  "date_joined": "…",
  "is_broker": true,
  "is_developer": false,
  "is_admin": false,
  "broker":   { "phone_number": "+79990001122", "inn_number": "…", … },
  "developer": null,
  "documents": [ … ]
}
```

**Ошибки:**
- `400` — пустой payload, невалидный ИНН, коллизия email/ИНН
- `401` — нет JWT
- `403` — пользователь помечен `is_active=false`
- `409` — race на unique-constraint (email/inn_number уже занят)

---

## 3. `PATCH /api/v1/admin/users/{id}/` (admin-update)

**Доступ:** `IsAdminUser`. Остальные — 403.

**Дополнительно у админа:** поле `is_active` для блокировки/разблокировки
пользователя. Админ **не может** деактивировать свой собственный аккаунт —
получит 400.

**Пример — админ меняет компанию девелопера и блокирует аккаунт:**

```http
PATCH /api/v1/admin/users/42/
Authorization: Bearer <jwt>
Content-Type: application/json

{ "company_name": "ООО Новое имя", "is_active": false }
```

**Ответ 200:**

```json
{
  "message": "Пользователь успешно обновлён.",
  "user": { /* TokenUserSerializer */ }
}
```

**Ошибки:**
- `400` — пустой payload, валидация, попытка деактивировать себя
- `403` — не админ
- `404` — пользователь не найден
- `409` — конфликт уникальных полей

---

## 4. Связь с уже существующими endpoint'ами

- `PATCH /admin/developers/{id}/` — **остаётся**, но покрывает только девелоперов. Новый `/admin/users/{id}/` — универсальный (броker + developer + admin).
- `PATCH /admin/users/{id}/block/` — **остаётся** для обратной совместимости (только `is_active`). Новый endpoint делает то же самое и больше.
- `POST /auth/change-password/` — пароль меняется по-прежнему только здесь.

Фронт может постепенно мигрировать на универсальный `/admin/users/{id}/`.

## 5. Что поменять на фронте

- [ ] экран «Мой профиль» для всех ролей (broker/developer/admin) — форма с
      полями `first_name`, `last_name`, `email` + роль-специфичные (телефон у
      броker'а, company_name у девелопера, ИНН у броker'а).
- [ ] обработка ошибок 400/409 с подсветкой поля (email или ИНН уже занят).
- [ ] в админке — экран редактирования пользователя любой роли, с переключателем
      `is_active` (плюс UI-охрана «нельзя деактивировать себя»).
- [ ] смена пароля — отдельный модал, вызывает существующий
      `/auth/change-password/`.

## 6. Вопросы для продукта

- Разрешить ли админу менять `role` (броker ↔ developer)? Сейчас **нет** —
  слишком много побочных эффектов (нужно создавать/удалять Broker/Developer).
- Триггерить ли ре-верификацию броker'а при смене `inn_number`? Сейчас **нет**,
  просто меняем поле.
- Нужны ли webhook'и / уведомления пользователю о том, что его профиль
  изменил админ? Сейчас **нет**.
