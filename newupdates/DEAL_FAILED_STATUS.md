# ТЗ 8.1 — Автоматический статус «Несостоявшийся» для сделки

Изменения фронта, необходимые после добавления нового статуса сделки `FAILED`.

---

## 1. Что изменилось в бэкенде (в двух словах)

- У `Deal.status` появилось новое значение: **`failed`** (отображается как «Несостоявшийся»).
- Celery-задача `deals.tasks.mark_failed_pending_deals` запускается **раз в сутки** и переводит сделки из `pending_documents` в `failed`, если они «висят» в этом статусе более **5 дней** (настраивается в `DEAL_PENDING_DOCUMENTS_FAIL_DAYS`).
- При переводе в `failed`:
  - `obligation_status` становится `overdue`;
  - создаётся `DealLog` с `action = marked_failed`;
  - отправляются websocket/REST-уведомления брокеру, девелоперу и всем админам (`event_type = deal_failed`).
- Статус **терминальный**: сделка не может быть восстановлена ни через UI, ни через API. Чтобы продать объект, девелопер должен создать новый аукцион (сам объект остаётся в статусе `published`, поскольку SOLD он получает только после `developer_confirmed`).

---

## 2. Что поменять на фронте

### 2.1. Новое значение `deal.status`

Во всех местах, где фронт рендерит `deal.status`, добавьте маппинг:

| API value   | Подпись (RU)       | Рекомендуемый цвет/стиль               |
| ----------- | ------------------ | -------------------------------------- |
| `failed`    | Несостоявшийся     | тёмно-серый / красный outline бейдж    |

Старые значения (`pending_documents`, `admin_review`, `developer_confirm`, `confirmed`) не меняются.

### 2.2. Доступные действия в статусе `failed`

В статусе `failed` **все кнопки действий должны быть скрыты/заблокированы**:

- загрузка ДДУ — недоступна
- загрузка подтверждения оплаты — недоступна
- редактирование комментария брокера — недоступно
- «Отправить на проверку» — недоступно
- админская «Одобрить / Отклонить» — недоступна
- девелоперская «Подтвердить / Отклонить» — недоступна

Бэкенд в любом случае вернёт 400, но UI не должен показывать кнопки, чтобы не путать пользователя.

Рекомендуется показать баннер / блок с текстом вида:

> Сделка автоматически переведена в статус «Несостоявшийся», так как документы не были загружены в течение 5 дней. Восстановить сделку нельзя. Обратитесь к администратору, если считаете это ошибкой.

### 2.3. Фильтры в списках сделок

В `/deals` есть фильтр по `status`. Добавьте новое значение `failed` в селектор статуса:

- для брокера — «Мои несостоявшиеся сделки»
- для девелопера — «Несостоявшиеся сделки по моим объектам»
- для админа — отдельный фильтр + возможно отдельная вкладка/счётчик

### 2.4. Новое событие в ленте уведомлений

В WebSocket / REST-уведомлениях появится новый `event_type`:

```
event_type = "deal_failed"
category   = "deal"
data = {
    "deal_id": <int>,
    "property_id": <int>,
    "broker_id": <int>,          // только в уведомлениях для девелопера и админа
    "days_in_pending": <int>,    // сколько дней сделка простояла в pending_documents
}
```

Тексты, которые бэкенд уже кладёт в `message`:

- **Брокеру:**
  `Сделка по <адрес> признана несостоявшейся: документы не были загружены в течение N дней`
- **Девелоперу:**
  `Сделка по <адрес> признана несостоявшейся: брокер <ФИО> не загрузил документы в течение N дней. Объект снова доступен для размещения`
- **Админу:**
  `Сделка #<id> по <адрес> автоматически помечена как несостоявшаяся. Брокер <ФИО> не загрузил документы`

Иконку / цвет для этого события в UI желательно сделать таким же, как для `obligation_overdue` (красный/предупреждающий).

### 2.5. Лог сделки (`/deals/{id}/logs/`)

В списке `logs` появится новая запись с `action = "marked_failed"`. Добавьте её в маппинг человекочитаемых названий:

| action           | Подпись (RU)                              |
| ---------------- | ----------------------------------------- |
| `marked_failed`  | Автоматически помечена как несостоявшаяся |

Существующая запись `marked_overdue` остаётся и может прийти **до** `marked_failed` (обязательство просрочено → через несколько дней сделка помечена как несостоявшаяся).

### 2.6. Объект недвижимости

Бизнес-решение: при FAILED **объект не переносится в SOLD**. Если девелопер хочет продать его снова, он создаёт новый аукцион (поведение как для любого `published` объекта). На фронте никаких специальных действий не требуется — объект доступен как обычно.

## 4. Вопросы для продукта, если потребуется

- Нужен ли ручной override для админа (кнопка «вернуть в pending_documents»)? Сейчас нет, статус терминальный.
- Нужно ли автоматически создавать новый аукцион по объекту после FAILED? Сейчас нет, решает девелопер вручную.
- Порог 5 дней жёстко задан в настройке `DEAL_PENDING_DOCUMENTS_FAIL_DAYS`; менять через БД нельзя — только через деплой.
