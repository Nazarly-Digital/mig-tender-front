# ТЗ 8.3 — Запрос документов у брокера до принятия решения по аукциону

Новая фича: владелец аукциона (девелопер) или админ может запросить у
брокера-участника дополнительные документы **до** подтверждения результата
аукциона. Брокер загружает файлы в ответ. Этот обмен нужен, чтобы девелопер
мог осознанно принять решение о confirm/reject (см. ТЗ 8.2).

---

## 1. Модель `DocumentRequest`

```jsonc
{
  "id": 42,
  "auction": 10,
  "broker": 7,
  "broker_email": "broker@example.com",
  "requested_by": 3,
  "requested_by_email": "dev@example.com",
  "description": "Пришлите скан паспорта",
  "broker_comment": "Прикладываю",              // опциональный текст от брокера
  "status": "pending",                          // pending | answered | cancelled
  "response_documents": [
    { "id": 100, "file": "https://…/…/passport.pdf", "uploaded_at": "2026-04-16T12:34:56Z" }
  ],
  "created_at": "…",
  "updated_at": "…",
  "answered_at": null
}
```

Статусы:

| status      | Подпись (RU)    | Описание                                    |
| ----------- | --------------- | ------------------------------------------- |
| `pending`   | Ожидает ответа  | Запрос создан, брокер ещё не загрузил файлы |
| `answered`  | Отвечен         | Брокер загрузил файлы                       |
| `cancelled` | Отменён         | Сейчас не используется (зарезервирован)     |

---

## 2. Endpoint'ы

### 2.1. Создать запрос документов

```http
POST /api/v1/auctions/{id}/request-documents/
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "broker_id": 7,
  "description": "Пришлите скан паспорта и ИНН"
}
```

**Доступ:** `auction.owner` (девелопер) или `admin`.

**Валидация:** broker_id должен принадлежать участнику аукциона (у него должна
быть ставка в этом аукционе). Description не пустой, до 4000 символов.

**Ответ 201:** объект `DocumentRequest` в формате, описанном выше.

**Ошибки:**
- `400` — broker не участник / пустое описание
- `401` — нет JWT
- `403` — не владелец и не админ
- `404` — аукцион не найден

**Эффект:** брокер получает уведомление `documents_requested`.

### 2.2. Получить список запросов по аукциону

```http
GET /api/v1/auctions/{id}/document-requests/
Authorization: Bearer <jwt>
```

**Доступ:**
- Владелец аукциона и админ — видят все запросы по аукциону.
- Брокер — видит только свои запросы.
- Остальные — пустой список.

**Ответ 200:** массив `DocumentRequest` (без пагинации).

### 2.3. Загрузить ответ на запрос (брокер)

```http
POST /api/v1/auctions/document-requests/{id}/upload/
Authorization: Bearer <jwt>
Content-Type: multipart/form-data

files[]=@passport.pdf
files[]=@inn.pdf
broker_comment=Прикладываю сканы
```

**Доступ:** только брокер, которому адресован запрос.

**Валидация:** 1–20 файлов, запрос должен быть в статусе `pending`.

**Ответ 200:** обновлённый `DocumentRequest` (status=`answered`,
`answered_at` заполнен, `response_documents` с url'ами загруженных файлов).

**Ошибки:**
- `400` — нет файлов / запрос уже отвечен или отменён
- `401`, `403`, `404`

**Эффект:**
- `status = answered`, `answered_at = now()`
- Девелопер (requested_by) получает уведомление `documents_request_answered`

## 3. Уведомления

Новые `event_type`:

| event_type                      | Кому            | data                                                                |
| ------------------------------- | --------------- | ------------------------------------------------------------------- |
| `documents_requested`           | Broker          | `{ auction_id, document_request_id, requested_by_id }`              |
| `documents_request_answered`    | Developer/Admin | `{ auction_id, document_request_id, broker_id, file_count }`        |

Категория — `auction`. Добавьте иконку/подпись в ленте уведомлений.

## 4. Что поменять на фронте

### 4.1. Страница аукциона (девелопер/админ)

В блоке «Ставки» или «Финалисты» рядом с каждым участником показать кнопку
**«Запросить документы»**:

- открывает модалку с полем `description` (textarea, обязательное)
- на submit → `POST /auctions/{id}/request-documents/` с `broker_id` и `description`
- после успеха — показать баннер «Запрос отправлен», обновить список запросов

Рядом с аукционом / в отдельной вкладке «Запросы документов» показывать список
через `GET /auctions/{id}/document-requests/`. Для каждого запроса:

- описание, брокер, дата
- статус бейдж
- если `answered` — список файлов (ссылки) и `broker_comment`
- если `pending` — плашка «Ожидает ответа брокера»

### 4.2. Страница аукциона (брокер)

Если у брокера есть входящий запрос (status=`pending`, `broker == me`):

- на видном месте на карточке аукциона — блок «Запрос документов» с
  `description` и кнопкой «Загрузить ответ»
- клик → модалка с drag-n-drop для файлов (до 20 шт.) + опциональный
  `broker_comment`
- submit → `POST /auctions/document-requests/{id}/upload/` (multipart)
- после успеха — блок переходит в `answered`, показывает загруженные файлы

Если брокер уже ответил (status=`answered`) — показывать как read-only запись с
файлами, без возможности повторной загрузки.

### 4.3. Карточка аукциона перед confirm/reject (связь с 8.2)

Если по аукциону есть хотя бы один `pending`-запрос, в UI рядом с кнопками
«Подтвердить результат» / «Отклонить результат» показать предупреждение:

> У вас есть незакрытые запросы документов. Дождитесь ответа брокера или
> всё равно примите решение.

Кнопки при этом остаются активными — решение пользователя.

### 4.4. Уведомления

- `documents_requested` → для брокера: иконка «доки», ссылка на аукцион
- `documents_request_answered` → для девелопера: ссылка на аукцион + запрос
