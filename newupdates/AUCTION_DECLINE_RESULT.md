# ТЗ 8.5 — Отказ от текущего победителя аукциона (OPEN и CLOSED)

Новая фича: девелопер может отказаться от текущего победителя аукциона
и **автоматически** переключиться на следующего кандидата. Это отличается от
ТЗ 8.2 (`/reject-result/`), где отказ всегда терминальный.

---

## 1. Отличие от ТЗ 8.2

| endpoint           | что делает                                                      | когда использовать                          |
| ------------------ | --------------------------------------------------------------- | ------------------------------------------- |
| `/reject-result/`  | помечает аукцион `failed` сразу, без поиска следующего          | «ни один из вариантов не подходит»          |
| `/decline-result/` | ищет следующего кандидата, `failed` только если очередь пуста   | «этот не подходит, покажите следующего»     |

---

## 2. Endpoint

```http
POST /api/v1/auctions/{id}/decline-result/
Authorization: Bearer <jwt>
Content-Type: application/json

{ "reason": "Брокер не выходит на связь" }
```

**Доступ:** владелец аукциона (девелопер) или админ.

**Предусловия:**
- `auction.status == finished`
- `auction.winner_bid != null`
- если для текущего `winner_bid` уже создана сделка, её `Deal.status` должен
  быть `pending_documents`. Если сделка ушла в `admin_review` /
  `developer_confirm` / `confirmed` — отказ запрещён, возвращается **400**.

**Ответ 200:**

```json
{
  "auctionId": 42,
  "status": "finished",
  "ownerDecision": "pending",
  "auctionFailed": false,
  "newWinnerBidId": 77
}
```

или (если кандидатов больше нет):

```json
{
  "auctionId": 42,
  "status": "failed",
  "ownerDecision": "rejected",
  "auctionFailed": true,
  "newWinnerBidId": null
}
```

**Ошибки:**
- `400` — нет `winner_bid`, пустой `reason`, сделка в не-PENDING-статусе
- `401`, `403`, `404` — стандартные

---

## 3. Алгоритм подбора следующего кандидата

### OPEN
Среди всех ставок аукциона, **исключая уже отклонённые** (`auction.declined_bids`),
выбирается ставка с максимальным `amount` (тай-брейкер — самая ранняя
`created_at`).

### CLOSED
- Если `shortlisted_bids` непустой после удаления отклонённого — выбирается
  следующая шорт-листовая ставка с максимальным `amount`.
- Если шорт-лист исчерпан — выбор идёт по всем запечатанным ставкам аукциона
  (excluding declined).
- Если никого — аукцион переводится в `failed`.

После выбора нового кандидата:
- `auction.winner_bid` = новый
- `auction.owner_decision` = `pending` — владелец должен заново подтвердить
  (`/confirm-result/`) или опять отказаться (`/decline-result/` или
  `/reject-result/`)
- для CLOSED: новый кандидат добавляется в `shortlisted_bids`

---

## 4. Что происходит со сделкой

Если на момент отказа уже была создана сделка для отклонённой ставки
(`Deal.status = pending_documents`):

- `Deal.status` → `declined` (новый статус, см. ТЗ 8.1 для сравнения с `failed`)
- в `DealLog` создаётся запись с `action = marked_declined`

Сделка **не удаляется** — остаётся в БД для аудита. Для фронта это значит,
что у брокера может появиться сделка в новом статусе `declined`.

Новый статус сделки:

| API value  | Подпись (RU)                 | Описание                                       |
| ---------- | ---------------------------- | ---------------------------------------------- |
| `declined` | Отклонена девелопером        | Девелопер отказался от результата аукциона     |

Новая запись в логе:

| action            | Подпись (RU)                                       |
| ----------------- | -------------------------------------------------- |
| `marked_declined` | Отклонена девелопером (отказ от результата аукциона)|

---

## 5. Уведомления

Два новых `event_type`:

| event_type                 | Кому              | data                                               |
| -------------------------- | ----------------- | -------------------------------------------------- |
| `auction_winner_declined`  | Declined broker   | `{ auction_id, declined_bid_id, reason }`          |
| `auction_winner_promoted`  | New winner, admins| `{ auction_id, winner_bid_id, broker_id? }`        |

Если очередь кандидатов пуста и аукцион стал `failed`, также отправляется уже
существующий `auction_result_rejected` (см. ТЗ 8.2).

Тексты бэкенда:
- **Declined broker:** `Девелопер отказался от вашей ставки по аукциону #N. Причина: …`
- **New winner:** `Вы стали победителем аукциона #N после отказа предыдущего кандидата. Дождитесь подтверждения от девелопера`
- **Admins (promoted):** `В аукционе #N новый победитель: <ФИО>. Ожидается решение девелопера`

## 6. Что поменять на фронте

### 6.1. Кнопка «Отказаться» на карточке аукциона (owner/admin)

На экране результата FINISHED-аукциона (там же где уже есть кнопки
«Подтвердить»/«Отклонить» из ТЗ 8.2) добавьте третью кнопку:

- **«Отказаться от кандидата»** — открывает модалку с обязательным полем
  причины, затем `POST /decline-result/`.
- Показывать кнопку только если:
  - `auction.status == finished`
  - `auction.owner_decision == pending`
  - `auction.winner_bid != null`
  - у текущего winner либо нет сделки, либо `Deal.status == pending_documents`.
    Если сделка ушла дальше — кнопку задизейблить с тултипом «Сделка уже на
    следующем этапе; используйте соответствующие кнопки отклонения».

Полезно визуально развести три кнопки:

| кнопка                        | что делает                             |
| ----------------------------- | -------------------------------------- |
| Подтвердить результат         | `/confirm-result/` → создаёт сделку    |
| Отказаться от кандидата       | `/decline-result/` → следующий в очереди|
| Отклонить результат аукциона  | `/reject-result/` → аукцион `failed`   |

### 6.2. Обработка ответа

- `auctionFailed == true` → показать, что аукцион теперь несостоявшийся; больше
  кнопок не показывать.
- `auctionFailed == false` → перезагрузить карточку аукциона, показать
  нового победителя (`newWinnerBidId`), снова в состоянии «ожидает решения».

### 6.3. Состояние участников

- Брокер, которого отклонили, видит статус «Отклонён» у своей ставки и
  получает `auction_winner_declined` в ленте.
- Брокер, ставший победителем, видит статус «Ожидает подтверждения девелопера»
  и получает `auction_winner_promoted`.

### 6.4. Список сделок брокера

Добавьте `declined` в enum `DealStatus` и локализацию. Иконка/цвет — как у
`failed`, но текст другой.

### 6.5. Лог сделки

В маппинге действий добавьте:

| action            | Подпись (RU)                                    |
| ----------------- | ----------------------------------------------- |
| `marked_declined` | Девелопер отказался от результата аукциона     |
